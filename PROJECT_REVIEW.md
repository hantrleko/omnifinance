# OmniFinance 项目评估（快速体检）

## 总体结论

项目方向清晰、功能覆盖完整，已经具备“可演示、可使用”的个人理财工具形态。当前最值得优先补强的是：

1. **工程结构去重**（`/` 与 `integrated_app/` 基本重复）；
2. **计算逻辑与 UI 解耦**（提升可测试性）；
3. **补齐基础自动化测试与 CI**（保证后续迭代稳定）。

## 已有优势

- 使用 Streamlit 多页面承载 7 个金融场景，产品定位直观，学习与演示价值高。
- 每个页面功能完整，具备图表、指标卡、导出等实用能力。
- 依赖集简洁，启动门槛低（`streamlit + pandas + plotly + yfinance`）。

## 主要改进点

### 1) 目录重复导致维护成本高

当前根目录与 `integrated_app/` 下存在同构文件集，后续需求变更容易发生“双份修改不一致”。

**建议**：保留单一应用入口，另一份改为文档说明或软链接/脚本生成，避免并行维护。

### 2) 业务计算与页面展示耦合较深

例如复利、回测逻辑都直接写在页面脚本中，虽然开发速度快，但单元测试与复用难度较高。

**建议**：将核心函数下沉到 `core/` 模块（如 `core/compound.py`, `core/backtest.py`），页面仅负责参数收集和展示。

### 3) 自动化测试与持续集成已接入，建议继续加强

当前已具备 `pytest` 测试与 GitHub Actions CI（含语法检查与测试执行），工程基线明显提升。

**建议**：下一步扩展覆盖率（边界值、异常输入、更多金融场景）并增加门禁策略（如覆盖率阈值/必过检查）。

### 4) 数据源容错可进一步加强

行情页与回测页对外部数据源依赖较高；网络波动时可读性会下降。

**建议**：增加“最近一次成功缓存”展示与错误提示分级（网络错误 / 代码不存在 / 限流）。

## 推荐优先级路线图（两周）

- **P0（本周）**：
  - 清理重复目录，统一唯一代码源；
  - 抽离核心计算模块；
  - 增加最小测试集。
- **P1（下周）**：
  - 提升 CI 门禁强度与测试覆盖率；
  - 统一日志与错误提示；
  - 增加 README 的运行、截图、免责声明。
- **P2（可选）**：
  - 增加组合回测、手续费/滑点、参数网格搜索等进阶能力。

## 一句话建议

这个项目**产品方向是对的**，建议尽快从“功能可用”迈向“工程可维护 + 可验证”，这样后续扩展会非常顺畅。
